# SMS OTP via MSG91

## Overview

Replace console.log OTP delivery with MSG91's OTP API for production SMS delivery. MSG91 handles OTP generation, SMS sending, retry/voice fallback, and verification. In dev mode (no MSG91 credentials), fall back to current Redis-based OTP with console.log.

## Current Flow

1. `POST /auth/resident-otp` → generates 6-digit OTP → stores in Redis (key `otp:{phone}`, 300s TTL) → logs to console
2. `POST /auth/resident-verify` → checks Redis → deletes key → returns JWT
3. `POST /auth/resident-register` → generates OTP → stores in Redis (key `reg:{phone}` with JSON payload, 300s TTL) → logs to console
4. `POST /auth/resident-register-verify` → checks Redis → deletes key → creates resident → returns JWT

## New Flow

### Login (resident-otp / resident-verify)

1. `POST /auth/resident-otp`:
   - Validate phone, check resident exists (keep existing logic)
   - If MSG91 configured: call `msg91.sendOTP(phone)` — MSG91 generates and sends OTP
   - If MSG91 not configured: generate OTP, store in Redis, log to console (current behavior)
   - Return `{ message: 'OTP sent' }`

2. `POST /auth/resident-verify`:
   - If MSG91 configured: call `msg91.verifyOTP(phone, otp)` — MSG91 verifies
   - If MSG91 not configured: check Redis (current behavior)
   - On success: look up resident, sign JWT, return token + user

### Registration (resident-register / resident-register-verify)

Registration is different from login because we need to carry forward community/unit context between the two calls. MSG91 only stores the OTP, not our custom payload.

1. `POST /auth/resident-register`:
   - Validate community code, find unit, check phone not registered (keep existing logic)
   - Store registration context in Redis: key `reg:{phone}`, value `{community_id, community_name, unit_id, unit_number}`, TTL 300s — but WITHOUT the OTP field
   - If MSG91 configured: call `msg91.sendOTP(phone)` for SMS delivery
   - If MSG91 not configured: also store OTP in the Redis payload (current behavior)
   - Return `{ message: 'OTP sent', communityName }`

2. `POST /auth/resident-register-verify`:
   - If MSG91 configured: call `msg91.verifyOTP(phone, otp)` — MSG91 verifies
   - If MSG91 not configured: check OTP from Redis payload (current behavior)
   - On success: read registration context from Redis, delete key, create resident, return JWT

## MSG91 API Wrapper

New file: `services/api-gateway/src/lib/msg91.js`

```javascript
const MSG91_BASE = 'https://control.msg91.com/api/v5';
const authKey = process.env.MSG91_AUTH_KEY;
const templateId = process.env.MSG91_TEMPLATE_ID;
const otpLength = parseInt(process.env.MSG91_OTP_LENGTH || '6', 10);

function isConfigured() — returns true if MSG91_AUTH_KEY is set

async function sendOTP(phone) — POST to /otp with {template_id, mobile: `91${phone}`, otp_length}
  - Prepends country code 91 (India)
  - Throws on non-success response

async function verifyOTP(phone, otp) — POST to /otp/verify with {mobile: `91${phone}`, otp}
  - Returns true on success, false on failure
  - Does NOT throw on invalid OTP (returns false)
```

Uses `axios` (already a dependency) for HTTP calls. No new dependencies needed.

## Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `MSG91_AUTH_KEY` | No (dev works without) | — | MSG91 dashboard authkey |
| `MSG91_TEMPLATE_ID` | Only if AUTH_KEY set | — | DLT-registered OTP template ID |
| `MSG91_OTP_LENGTH` | No | `6` | OTP digit count |

## Dev Mode Behavior

When `MSG91_AUTH_KEY` is not set:
- OTPs are generated locally (current `Math.random` approach)
- Stored in Redis with 300s TTL
- Logged to console: `[DEV] OTP for {phone}: {otp}`
- No external API calls

This means local development and testing work without MSG91 credentials.

## Prerequisites (manual, outside this spec)

1. Create MSG91 account at msg91.com
2. Complete DLT registration (JioTrueConnect or similar)
3. Register sender ID (e.g., "CMGATE")
4. Register OTP template: "Your CommunityGate OTP is {#var#}. Valid for 5 minutes."
5. Get authkey + template ID from MSG91 dashboard
6. Set environment variables on EC2
